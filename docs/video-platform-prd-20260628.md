# 智选亿万用户级视频平台终局 PRD - 2026-06-28

## 1. 文档目标

本 PRD 用于定义 `智选` 视频系统从“当前可用的视频功能”升级为“亿万用户级商业视频平台”的终局目标，覆盖：

- 产品边界
- 架构边界
- 存储与分发策略
- 播放体验目标
- 成本控制原则
- 治理与观测闭环

本文件不是局部优化说明，而是平台级定版文档。

## 2. 产品终局定义

### 2.1 目标

打造一套可支撑亿万用户级日活、海量视频上传与高并发播放的商业级视频平台，满足以下核心诉求：

- 推荐流具备统一、极致、沉浸的视觉秩序
- 视频播放具备极低首帧延迟、极低卡顿率、极低失败率
- 存储和分发成本在超大规模下可控
- 平台具备完整的审核、治理、审计、生命周期管理能力
- 平台具备可观测、可回滚、可演进、可重建能力

### 2.2 产品形态

- `推荐`：仅承载 `9:16` 竖向视频流
- `横屏`：横向视频独立专区，不进入推荐主流
- `上传`：支持素材裁剪与方向选择，但最终发布标准由服务端统一收口
- `播放`：以沉浸、快速、稳定为第一优先级，不靠遮挡和提示掩盖真实播放状态

### 2.3 用户体验底线

- 推荐流不得混入横向视频
- 横向视频不得破坏主推荐流视觉秩序
- 视频切换不应依赖封面遮挡
- 播放卡顿时展示最后一帧真实视频，不展示伪装层
- 不允许通过降分辨率破坏主观清晰度

## 3. 当前系统与终局系统的本质差异

### 3.1 当前系统

当前系统已经具备以下能力：

- 端侧采集、裁剪、转码、上传
- Cloudflare R2 存储
- Worker 上传控制面
- Supabase 元数据入库
- 播放器池与预热策略
- HLS 主源 + MP4 兜底

但当前系统仍属于“产品能力闭环”，而不是“平台能力闭环”。

### 3.2 终局系统

终局系统必须从“单条上传播放链路”升级为“视频基础设施平台”，核心差异在于：

- 客户端不再承担最终发布转码标准
- 服务端统一生成唯一主分发资产
- 播放主链路不再长期依赖完整 MP4
- 存储采用冷层/热层分层治理
- 分发采用分片化流媒体而非整文件直拉
- 整个平台具备观测、治理、审计、重试、回滚、降本能力

## 4. 核心产品原则

### 4.1 推荐流原则

- 推荐流只允许 `9:16`
- 推荐流统一画布标准为 `1080 x 1920`
- 推荐流是平台最高等级体验区域
- 推荐流中的每一条视频都必须遵循统一构图秩序与统一播放标准

### 4.2 横屏原则

- 横屏视频存在价值，但不进入推荐主流
- 横屏视频单独在 `横屏` 频道或专区承载
- 横屏频道播放标准独立定义，不污染推荐体验

### 4.3 清晰度原则

- 平台不依赖“网络差就降分辨率”的粗暴策略
- 推荐流的目标空间分辨率固定为 `1080 x 1920`
- 在固定分辨率下，通过编码效率、内容自适应码率和预取调度优化成本与稳定性

### 4.4 播放原则

- 视频切换前，下一条必须尽可能完成首帧准备
- 播放区域只展示真实视频画面，不展示封面遮挡层
- 缓冲或卡顿时停留在最后成功渲染的一帧
- 平台追求“接近零可感知掉帧”，而不是依赖 UI 装饰掩盖问题

## 5. 终局架构蓝图

### 5.1 总体结构

系统拆分为以下 8 个层级：

1. 客户端采集与上传层
2. 上传签发与鉴权层
3. 编排与工作流层
4. 转码与打包层
5. 存储分层与对象治理层
6. 播放清单与边缘分发层
7. 客户端播放与预热层
8. 审核、治理、观测、成本控制层

### 5.2 客户端职责

客户端只负责：

- 相机采集或相册选取
- 轻量裁剪与交互确认
- 上传原片与元数据
- 播放平台生成的分发资产

客户端不负责：

- 平台最终标准转码
- 平台最终主分发资产生成
- 平台编码兼容策略决策

### 5.3 上传签发层

上传必须先进入上传签发层，而不是直接写对象存储。

上传签发层负责：

- 登录态鉴权
- 上传会话签发
- 对象前缀规划
- 文件指纹与幂等控制
- MIME 白名单校验
- 视频方向和业务归属标签校验
- 分片上传策略与续传策略下发

### 5.4 编排与工作流层

上传完成后，视频进入异步编排层。

工作流至少包含以下步骤：

1. 文件校验
2. 病毒和非法内容预扫描
3. 元数据提取
4. 冷母版生成
5. 热分发资产生成
6. 封面生成
7. 质量检测
8. 发布入库
9. CDN 预热
10. 审计与观测打点

每一步必须具备：

- 幂等
- 重试
- 失败隔离
- 可追踪
- 可人工重跑

### 5.5 转码与打包层

终局平台不再以“完整 MP4 直拉”作为主播放策略。

终局标准如下：

- 推荐流主资产：`1080 x 1920`
- 横屏主资产：`1920 x 1080`
- 主分发格式：`HLS + CMAF/fMP4`
- Segment 目标时长：`1s`
- GOP 目标：`1s`
- 封面：独立生成，服务于列表卡片，不进入播放面遮挡

### 5.6 存储分层

#### 冷层

冷层用于存储：

- 原始原片
- mezzanine 中间母版
- 审计或申诉所需保留素材

冷层目标：

- 成本最低
- 读频低
- 生命周期清晰

#### 热层

热层用于存储：

- 唯一线上主分发资产
- 播放清单
- init segment
- media segment
- 封面

热层不得长期保留“完整 MP4 + HLS 双主资产”。

### 5.7 播放与客户端层

客户端播放层必须具备：

- 常驻播放器池
- 当前/下一条/下二条预热
- 首帧优先级调度
- Manifest 提前获取
- Init segment 提前获取
- 首段提前获取
- 首帧提前解码
- 失败快速回退

### 5.8 CDN 与边缘层

边缘分发必须遵循：

- Manifest 短缓存
- Init/segment 强缓存
- 热门视频区域化升温
- 极低回源率
- 对象不可变发布策略

## 6. 编码策略

### 6.1 编码目标

目标不是“统一高码率”，而是“统一高感知画质”。

### 6.2 核心方法

- 固定空间分辨率
- 内容自适应码率
- 按复杂度分配码率预算
- 使用高压缩效率编码作为主方向
- 使用兼容编码作为受控兜底

### 6.3 平台策略

#### 推荐流

- 固定 `1080 x 1920`
- 不因网络差动态降低空间分辨率
- 通过更高效编码和缓存命中率来控制成本

#### 横屏流

- 固定 `1920 x 1080`
- 与推荐流完全隔离

### 6.4 帧率策略

平台不追求“所有视频一律 60FPS”。

推荐策略：

- 源为 `60FPS`：保留 `60FPS`
- 源为 `30FPS`：保留 `30FPS`
- 不进行低价值插帧

原因：

- 可减少无意义的编码开销
- 可减少无意义的存储与带宽开销
- 可避免为“看起来更高”牺牲整体稳定性

## 7. 观测与治理闭环

### 7.1 观测指标

平台必须实时采集以下指标：

- 上传成功率
- 转码成功率
- 转码平均耗时
- 发布平均耗时
- 首帧时间
- Segment 加载失败率
- 自动回退率
- 平均缓冲时长
- 卡顿率
- 掉帧率
- CDN 命中率
- 回源率
- 单视频平均存储成本
- 单视频平均分发成本
- 按区域/设备/频道拆分的表现

### 7.2 治理能力

平台必须具备：

- 内容审核
- 指纹去重
- 侵权处理
- 生命周期管理
- 失效重转码
- 热点缓存升温
- 低价值内容降本
- 审计事件追踪

## 8. 商业级容量目标

### 8.1 平台必须支持

- 海量并发上传
- 海量并发播放
- 热点视频暴涨
- 大促或活动尖峰流量
- 区域性热点与跨区域扩散

### 8.2 架构必须具备

- 无单点
- 流水线可扩容
- 队列可削峰
- 转码可并行
- 边缘缓存可吸收读取高峰
- 任一子链路故障时不拖死整个平台

## 9. 技术选型原则

### 9.1 当前基础

当前项目已有以下底子：

- Flutter 客户端
- Supabase 数据与鉴权
- Cloudflare Worker
- Cloudflare R2

### 9.2 终局平台扩展方向

建议能力拆分为：

- `Supabase`：业务主数据、用户、身份、审核状态、对象元数据
- `Cloudflare R2`：对象存储冷热分层
- `Workers`：签发、鉴权、播放清单控制、边缘逻辑
- `Queues / Workflows`：转码编排、任务治理、失败重试
- `Analytics / Observability`：分发、性能、成本观测

## 10. 与当前系统的硬切换原则

平台不应再以“继续在旧主链路上补功能”为主要方向。

必须进行以下重构级升级：

- 客户端退出最终标准化转码责任
- 服务端建立唯一主分发资产生产线
- 公开播放链退出长期 MP4 双存
- 存储按冷层/热层重建
- 观测与治理系统正式立项

## 11. 验收标准

平台完成终局升级后，应满足：

- 推荐流只承载竖向视频
- 横屏视频与推荐完全隔离
- 主播放资产唯一化
- 分片流成为默认播放主链路
- 冷层与热层明确分层
- 平台拥有成本模型和实时观测
- 平台拥有治理与生命周期系统
- 播放首帧、卡顿、回退、失败数据可量化
- 平台具备继续向亿万用户规模放大的工程基础

## 12. 本文结论

`智选` 视频系统的下一步不是“继续修视频页”，而是“建设一套视频基础设施平台”。

终局目标已经明确：

- 推荐流竖向统一
- 横屏流独立承载
- 服务端统一转码
- 主分发资产唯一化
- 分片流主链路化
- 成本、治理、观测、审计一体化

这是一条平台升级路线，而不是单功能迭代路线。
